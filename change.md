● 推荐决策：把它做成 AI 驱动的“结构迁移系统”，不是让 AI 直接改分类表。
  也就是说，AI 可以自由提出、自由规划、在低风险范围内自由执行结构调整；但真正落盘必须走一条可模拟、可回滚、可审计的迁移链路。
  这不是彻底重构整个系统，但也不是小补丁。更准确说，它是一次 中层架构升级：外层 /wiki 和 wiki 基本保留，内层补上 身份层 + 迁移层 + 维护闭环。
  先说最关键判断
  当前底座已经证明了两件事：
   1. 轻量分类骨架是对的：domain / primary_type / subtype / tags + registry + suggestion queue
   2. AI/CLI 分工也是对的：AI 判断，CLI 确定性执行
  但现在有一个决定性的短板：
  系统还把 target_page / 文件路径 当成“对象身份”在用。
  这会导致：
   - 改路径 = 改身份
   - 改分类 = 连带改 path / index / proposal / cross_ref
   - merge / split / rename 一旦发生，就不是“更新分类”，而是“搬整个系统”
  所以，如果你已经确定“结构一定会调整”，那下一步最重要的不是继续补 taxonomy 字段，而是：
  把“知识对象身份”从“文件路径”里解耦出来。
  ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  最小优雅模型
  我建议把系统正式分成 4 层：
  ┌────────────────────┬─────────────────────────────────────────────────────┬──────────────┐
  │ 层                 │ 作用                                                │ 是否稳定     │
  ├────────────────────┼─────────────────────────────────────────────────────┼──────────────┤
  │ Identity           │ 知识对象的稳定身份，如 page_id                      │ 稳定         │
  ├────────────────────┼─────────────────────────────────────────────────────┼──────────────┤
  │ Classification     │ domain / primary_type / subtype / tags              │ 半稳定       │
  ├────────────────────┼─────────────────────────────────────────────────────┼──────────────┤
  │ Placement          │ 当前放在哪个目录、哪个 collection、哪个索引页       │ 可调整       │
  ├────────────────────┼─────────────────────────────────────────────────────┼──────────────┤
  │ Governance         │ suggestion、migration plan、review、apply、rollback │ 流程稳定     │
  └────────────────────┴─────────────────────────────────────────────────────┴──────────────┘
  这 4 层必须分开
  1. Identity 层
  这是知识对象本身，不应该因为结构调整而变化。
  建议新增：
   - page_id
   - source_id
   - proposal_id
  其中最关键的是 page_id。
  以后页面的真正身份应该是 page_id，不是 path。
  ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  2. Classification 层
  这是“它是什么”。
  保留：
   - domain
   - primary_type
   - subtype
   - tags
  并继续维持：
   - registry
   - suggestion queue
  这层可以被 AI 持续优化，但不能无控制直接改写。
  ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  3. Placement 层
  这是“它现在放哪”。
  这里要特别注意：
  你现在路径里的中间段，比如：
   - ai/rag/chunk-size-strategy
   - ai/databases/vector-db-comparison
   - ai/decisions/finetuning-vs-rag
  这个中间层很多时候不是 semantic type，而是导航结构 / topic shelf。
  所以建议显式引入一个概念：
   - collection 或 navigation_bucket
  这样以后：
   - primary_type=guide
   - subtype=workflow
   - collection=rag
  如果未来想从 rag/ 搬到 retrieval/，只是 placement 变化，不是 semantic taxonomy 崩掉。
  这是非常关键的架构解耦。
  ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  4. Governance 层
  这是 AI 能不能“自由调整但不失控”的根本。
  必须新增一条专门的结构治理链，不要把它塞回现在的 content proposal。
  ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  最重要的子决策：不要把结构重组混进现有 change-proposal
  当前 change-proposal 很适合：
   - create
   - update
   - archive
  它处理的是内容编译。
  但你要支持的是：
   - rename category
   - merge subtype
   - split domain
   - reclassify page
   - relocate page
   - deprecate taxonomy item
  这些不是内容编译，是结构迁移。
  所以最合理的做法是：
  保留两条独立控制面
  ┌──────────────────────────────────────┬──────────────────────────────────────────────────┐
  │ 控制面                               │ 负责什么                                         │
  ├──────────────────────────────────────┼──────────────────────────────────────────────────┤
  │ content proposal                     │ 页面内容的 create/update/archive                 │
  ├──────────────────────────────────────┼──────────────────────────────────────────────────┤
  │ migration plan / taxonomy change     │ rename/merge/split/reclassify/relocate/deprecate │
  └──────────────────────────────────────┴──────────────────────────────────────────────────┘
  不要混。
  一旦混在一起：
   - compile 会变复杂
   - proposal 语义会混乱
   - AI prompt 会越来越不稳定
   - review 和 apply 的风险边界会失真
  ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  我建议的新核心对象
  最少要新增这几类：
  1. Page
   - page_id
   - current_path
   - title
   - classification
   - placement
   - status
   - aliases
   - source_refs
  2. Taxonomy Registry
   - domain
   - primary_type
   - subtype
   - aliases
   - status
   - version
  3. Migration Plan
  用于结构变化，不是内容变化。
  字段建议至少有：
   - plan_id
   - operation_type
   - scope
   - from
   - to
   - affected_page_ids
   - path_changes
   - taxonomy_changes
   - risk_level
   - dry_run_report
   - rollback_plan
   - status
  4. Alias / Redirect Map
  这个非常重要。
  结构调整后，旧路径不能立刻“消失”，否则：
   - 历史 proposal 断掉
   - cross refs 断掉
   - 外部引用失效
   - query 命中抖动
  所以要有：
   - old_path -> page_id
   - old_taxonomy_value -> new_taxonomy_value
  ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  AI 应该怎么用这套系统
  正确方式不是“AI 随手改分类文件”。
  而是这条闭环：
  1. Observe
  maintain 发现结构问题：
   - 某 subtype 过大
   - 某 subtype 只有 1 个页面
   - suggestion 长期积压
   - query 候选集过大
   - 跨域 cross_refs 密度异常
   - 同一页被频繁改分类
  2. Propose
  AI 产出结构迁移方案：
   - 合并哪些 subtype
   - 废弃哪些分类
   - 哪些页面要搬
   - 哪些 collection 要重命名
  3. Simulate
  CLI 做 dry-run：
   - 影响多少 page
   - 影响多少 proposal
   - 哪些 index 会重写
   - 哪些 old path 需要 alias
   - 哪些 cross_refs 会断
   - query candidate size 是变好还是变坏
  4. Decide
  按风险级别决定：
   - 低风险：AI 可自动执行
   - 中风险：AI 自动生成计划，系统摘要通知
   - 高风险：抬给人一个最小确认问题
  5. Apply
  CLI 确定性执行迁移：
   - 改 registry
   - 改 classification assignment
   - 改 placement/path
   - 写 alias
   - 重建 index
   - 更新 runtime db
   - 记 migration log
  6. Verify
  迁移后再跑维护评分：
   - 候选集是否更小
   - suggestion 是否更少
   - orphan 是否减少
   - 冲突是否变少
   - query 命中是否更稳
  7. Rollback
  如果效果更差，按 plan rollback。
  ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  “调整后是否能得到最合理的最新分类”靠什么保证
  不能靠 AI 主观觉得“更优”，必须靠维护指标。
  我建议 maintain 以后增加 5 类信号：
   1. Overflow：某分类页数过多，应拆分  
   2. Sparsity：某分类长期只有 0-1 页，应合并  
   3. Ambiguity：同一值在多个上下文高频出现，边界不清  
   4. Drift：pending/rejected suggestion 持续增加，说明 taxonomy 不稳  
   5. Retrieval inefficiency：query 经常命中过大的候选集，说明分类不能有效缩圈
  这样“最合理的最新分类”不是拍脑袋，而是：
  AI 发现候选结构 + maintain 用指标验证 + migration engine 安全执行。
  ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  这套底座支不支持
  支持。
  原因是现在已经有：
   - registry
   - suggestion queue
   - runtime index
   - deterministic CLI
   - maintain/check/review/apply 的治理习惯
   - 可重建的 SQLite 索引
  这些都说明底座方向对了。
  但必须补一个关键升级
  引入 page_id，并把 identity / classification / placement 分开。
  这是唯一一个不能再拖的中层改造。
  如果不做，后面每次结构调整都会越来越像“手术拆楼”。
  所以我的判断是：
   - 不是推翻底座
   - 但必须做一次关键中层重构
   - 重构点不是检索，不是 embedding，不是 graph
   - 重构点就是：身份解耦 + 迁移控制面
  ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  推荐的落地顺序
   1. 先加 page_id
   2. 再加 migration-plan schema
   3. 再做 dry-run / impact analysis
   4. 再做 alias / redirect
   5. 最后让 maintain 生成结构优化候选
  这个顺序不能反。
  如果还没 page_id 就先做 merge/remove/move，后面一定乱。
  ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  最终建议
  最优解不是“AI 自由改 taxonomy”，而是“AI 自由驱动 taxonomy migration”。
  一句话概括：
   保留现在的轻量分类底座，不走重图谱；新增稳定 identity、可变 placement、版本化 taxonomy 和 migration control plane，让 AI 有自由调整结构的能力，但所有结构变化都必须被模拟、审计、验证、回滚。
