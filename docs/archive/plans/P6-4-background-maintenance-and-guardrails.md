# P6-4：后台维护与内部 Guardrail 方案

## 背景

如果前台要做到“一个入口、低认知负担”，就必须把大量系统性动作收进后台。

但后台化不能等于“失控自动化”。  
因此，P6 需要同时解决两件事：

1. **哪些能力应该被系统吞到后台**
2. **后台自动化之后，如何继续保效果**

这份方案就是为了解决这两个问题。

---

## 目标

实现一个产品层面足够轻，但系统层面仍有约束的结构：

1. 前台少操作
2. 后台自动维护
3. 评测继续做内部 guardrail
4. 不因为简化体验而牺牲效果

---

## 应后台化的能力

以下能力不应再成为用户显式操作项：

### 1. lint

原定位：手动检查系统健康状态  
P6 定位：后台定时运行

### 2. maintain

原定位：手动维护 staleness、归档过期页面  
P6 定位：后台定时运行

### 3. 健康巡检

原定位：只有维护者主动检查  
P6 定位：系统自动巡检，必要时只给摘要

### 4. proposal schema 完整性检查

原定位：靠前台或评测才发现  
P6 定位：后台自动校验

---

## 后台自动化分层

### Layer 1：周期性后台任务

适合放进 CI / cron：

- lint
- maintain
- stale proposal 检查
- patrol 去重检查
- 健康度摘要生成

特点：

- 固定周期执行
- 不依赖用户触发
- 默认不打扰用户

### Layer 2：事件驱动后台任务

由用户操作触发，但不要求用户感知：

- query 后自动 gap/write-back 检查
- ingest 后自动 proposal 起草
- organize 后自动批量刷新

特点：

- 依附主流程
- 作为系统内部动作附带执行
- 结果只在“系统动作摘要”中简要呈现

### Layer 3：受控高风险动作

不进入全自动后台：

- 直接写 canon
- 冲突裁决
- 重要 proposal 审批
- 大规模关键页替换

特点：

- 保留确认门
- 不因后台化而失控

---

## 推荐的后台运行模型

### 日常模式

1. 用户正常使用单入口 Skill
2. 系统内部完成 query / ingest / organize
3. 后台自动做：
   - write-back 起草
   - schema 校验
   - 状态更新

### 定时模式

每天 / 每周后台执行：

1. lint
2. maintain
3. stale proposal 检查
4. 健康摘要输出

### 异常模式

当后台发现异常时，才向用户或维护者抬出：

- health score 明显下降
- 重复 patrol 增加
- stale backlog 持续增长
- proposal schema 异常频发

---

## 内部 Guardrail 设计

前台变简单后，质量保障必须留在后台。

因此，benchmark / rubric / protocol 不再作为用户接口，而是作为：

### 1. 内部回归护栏

每次重要规则调整后，继续用现有 frozen benchmark 检查：

- query 质量是否回退
- schema_valid 是否回退
- boundary_honesty 是否回退

### 2. 路由护栏

产品化后要特别检查：

- 单入口是否让 query 质量下降
- 自动 write-back 是否误触
- 默认输出是否把边界信息隐藏过度

### 3. 高风险动作护栏

继续检查：

- 直接影响 canon 的动作是否被确认门拦住
- 冲突 case 是否仍需要人工裁决
- 低置信内容是否未被误写入稳定结论

---

## 后台摘要设计

虽然用户不应手动操作后台维护，但系统仍应定期给出**低打扰摘要**。

### 推荐摘要形式

```text
本周系统状态：
- 已自动登记缺口：3 条
- 已发现待处理高风险变更：1 条
- 健康状态：稳定
- 需要你处理：1 个审批项
```

重点：

- 只给摘要
- 不把用户拉回技术细节
- 只有真正需要人决策时才抬出来

---

## 非目标

本阶段不追求：

1. 完全取消人工审查
2. 把 evaluation 全部自动公开展示给最终用户
3. 用 CI 取代前台 Skill 入口

---

## 成功标准

如果满足以下条件，就说明后台化方案合格：

1. 用户默认不再手动操作 lint / maintain
2. 后台维护不会让高风险动作失控
3. benchmark 继续作为内部 guardrail，前台效果不回退
4. 用户只在真正需要时才收到系统摘要或确认

---

## 当前结论

真正的“低认知负担”不是少做事，而是：

> **把必须做的复杂事留在系统后台，把必须给人的决策点压缩到最少。**

这份方案定义的，就是这层后台化边界。
