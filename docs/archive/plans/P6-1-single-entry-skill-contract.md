# P6-1：单入口 Skill 产品契约（effect-first / low-cognitive-load）

## 背景

前面阶段已经证明：当前知识治理内核在规则层面是有效的，但它仍然更像“后台系统”，而不是“前台好用产品”。

当前系统的问题不是能力不够，而是**用户操作面太复杂**：

- 用户容易被 `ingest / promote / compile / query / lint / maintain` 这些内部动作名拖入系统实现细节
- 用户需要理解 proposal、schema、write-back、benchmark、protocol 等概念
- 虽然系统效果已经做出来，但“怎么用它”仍然会给人带来额外认知负担

因此，P6 的第一步不是继续加功能，而是先把**单入口 Skill 的产品契约**定下来。

---

## 目标

把现有知识治理内核，包装成一个：

1. **单入口**
2. **低认知负担**
3. **效果优先**
4. **高风险动作仍可控**

的可用 Skill。

---

## 核心原则

### 原则 1：用户只看一个入口

用户不再被要求主动选择：

- query
- ingest
- reconcile
- maintain

这些是系统内部路由问题，不是用户操作问题。

### 原则 2：效果优先于透明度

默认优先保证：

- 回答质量
- 边界诚实
- 缺口自动登记
- 知识治理闭环

不是默认展示最多系统细节。

### 原则 3：复杂度默认隐藏，按需展开

默认模式只暴露最少信息。  
只有在用户主动要求时，才进入 audit / debug 视图。

### 原则 4：低风险自动化，高风险保留确认

应自动化的：

- 路由判断
- schema 填充
- write-back proposal 起草
- lint / maintain / 健康巡检

不应直接全自动的：

- canon 高风险写入
- 重要 proposal 审批
- 冲突裁决

---

## 推荐交付形态

### 最终推荐

**前台：单入口 Skill**  
**兼容：单命令 `/wiki` 外壳**  
**后台：现有 `.wiki + policy/specs + schemas` 内核**  
**后台维护：CI / cron**

### 不是主入口的东西

以下都不应该作为用户主入口：

- 多命令 CLI
- 多 Skill 手工选择
- prompt 词库
- CI pipeline

它们最多只能作为：

- 兼容层
- 内部实现层
- 后台维护层

---

## 用户输入契约

用户只需要提交**自然语言请求**或**原始资料**。

### 输入类型

| 类型 | 用户输入样例 | 系统内部主要路由 |
|---|---|---|
| 提问 | “什么时候应该把页面归档？” | query |
| 贴资料 | “请把这段会议纪要吸收进知识库” | ingest |
| 整理请求 | “帮我把这批内容整理并补缺口” | reconcile / refresh / maintain |
| 审计请求 | “告诉我这次用了哪些页面和 proposal” | audit view |

### 用户不需要显式输入的内容

以下内容默认不要求用户提供：

- `origin`
- `trigger_source`
- proposal 目录位置
- query / ingest / promote / compile 的动作名
- benchmark / protocol / rubric 细节

---

## 用户输出契约

## 默认模式

默认模式只输出 4 类信息：

1. **结果本身**
   - 回答 / 已吸收 / 已整理
2. **可信边界**
   - 哪些内容有 canon 依据
   - 哪些内容仍有缺口或争议
3. **系统动作摘要**
   - 是否自动登记了知识缺口
   - 是否触发了 proposal 起草
   - 是否触发了后台维护
4. **唯一需要用户决定的事项**
   - 如果没有高风险动作，就不额外打扰用户

### 默认模式示例

```text
结论：当前没有统一归档阈值。
依据：运营团队主张 90 天，研究团队主张 180 天。
系统动作：未自动写入 canon；已登记缺口补充建议。
需要你决定：无
```

## 审计模式

只有在用户主动要求时，才展开：

- 命中的 canon 页面
- 哪些句子属于外推断
- proposal 写入位置
- 内部路由结果
- 哪个规则拦截了高风险动作

---

## 系统内部契约

### 系统必须自动做的事

1. **意图路由**
   - 识别用户是在提问、贴资料还是请求整理
2. **边界控制**
   - 遇到无覆盖内容时，不把训练知识包装成确定结论
3. **缺口登记**
   - 发现知识缺口时，自动起草 write-back proposal
4. **schema 填充**
   - 自动补齐 `origin`、`trigger_source` 等字段
5. **后台维护**
   - 定时 lint / maintain / 健康巡检

### 系统不得自动做的事

1. 跳过高风险人工审查直接写 canon
2. 自动裁决高风险知识冲突
3. 在无依据时输出确定结论

---

## 人工确认边界

用户只在以下情况被要求确认：

1. **高风险 canon 改写**
2. **重要 proposal approve / reject**
3. **冲突裁决**
4. **低置信且可能影响现有结论的更新**

默认情况下，普通 query / 普通 ingest / 普通缺口登记不应该频繁打断用户。

---

## 内部路由总表（第一版）

| 用户表述 | 默认内部路由 | 用户是否需要理解 |
|---|---|---|
| “帮我回答” | query | 否 |
| “帮我吸收这段资料” | ingest | 否 |
| “帮我补这部分知识空缺” | query + write-back | 否 |
| “帮我整理这批知识” | reconcile / refresh / maintain | 否 |
| “给我看细节” | audit mode | 否（只在需要时打开） |

---

## 非目标

这一阶段不追求：

1. 让用户理解系统全部内部机制
2. 让用户直接操作所有后台命令
3. 把 CI 暴露成主入口
4. 取消所有人工确认

---

## 成功标准

如果满足以下条件，就说明这个产品契约是合格的：

1. 用户默认只面对一个入口
2. 用户默认不用理解 6 个内部 skill 的边界
3. 用户只靠自然语言就能完成问答 / 摄入 / 整理
4. benchmark 既有效果不因为产品化回退
5. 高风险动作仍然有安全边界

---

## 当前结论

这份契约定义了一个清晰方向：

> **用户前台极简，系统后台强治理。**

后续所有 P6 设计都应以此为准，而不是重新把用户拉回多命令、多协议、多系统概念的使用方式。
