# P6-2：单入口 Skill 内部路由设计

## 背景

在 P6-1 中，已经确定产品形态是：

- 用户前台只看到一个入口
- 系统内部自动判断该走哪条知识治理链路

因此，下一步必须把“自动路由”定义清楚。  
否则所谓单入口，最后只会退化成“看起来只有一个入口，实际上还是用户自己在猜系统要做什么”。

---

## 目标

为单入口 Skill 定义一套**稳定、低认知负担、效果优先**的内部路由规则，使系统能够在不额外打扰用户的前提下，自动选择最合适的治理路径。

---

## 路由原则

### 原则 1：先满足用户表层目标，再触发内部治理

用户表面上是在：

- 问问题
- 贴资料
- 让系统整理内容

系统首先必须满足这个外部目标。  
write-back、schema 填充、后台维护等都属于系统内部动作，不能反过来抢占用户目标。

### 原则 2：默认少打扰，必要时才确认

如果系统能安全自动完成，就不要中断用户。  
只有进入高风险区时，才向用户要确认。

### 原则 3：多意图请求允许内部拆分，不要求用户重说

若用户一句话里同时包含：

- 问答
- 摄入
- 整理

系统应尽量**内部拆分执行**，而不是要求用户先理解系统模块边界。

---

## 一级路由类型

单入口 Skill 内部只保留 4 条一级路由：

| 一级路由 | 用户表述范型 | 内部目的 |
|---|---|---|
| `answer` | “告诉我… / 帮我回答…” | query + 边界控制 |
| `absorb` | “请吸收这段资料 / 记到知识库里” | ingest + proposal 起草 |
| `organize` | “帮我整理 / 合并 / 补缺 / 更新这批内容” | reconcile / refresh / maintain |
| `audit` | “展开细节 / 给我看依据 / 看 proposal / 看路由过程” | audit view |

这 4 条一级路由由系统内部判断，**不要求用户显式选择**。

---

## 二级内部动作

一级路由选定后，再进入二级动作组合：

### 1. `answer`

可能触发：

- query
- 证据引用
- 冲突展示
- 缺口检测
- 自动 write-back proposal 起草（如有缺口）

### 2. `absorb`

可能触发：

- ingest
- source 文件生成
- proposal 起草
- 状态更新

### 3. `organize`

可能触发：

- reconcile
- refresh
- maintain
- batch write-back
- 健康巡检

### 4. `audit`

可能触发：

- 命中页面展开
- 外推断句子展开
- proposal 位置展开
- 路由解释展开
- 规则拦截原因展开

---

## 路由判定规则（第一版）

## Rule 1：显式审计意图优先

如果用户明确要求：

- “展开细节”
- “告诉我依据”
- “给我看 proposal”
- “给我看你怎么判断的”

则直接进入 `audit`，并在原主流程结果之上叠加 audit 展示。

## Rule 2：原始资料输入优先走 `absorb`

如果用户主要动作是：

- 贴长文本
- 提供会议纪要 / 文档 / 对话记录
- 明确说“吸收进知识库 / 入库 / 归档”

则优先走 `absorb`。

## Rule 3：明确问答意图优先走 `answer`

如果用户主要动作是：

- 提问
- 询问现有知识结论
- 让系统给出建议或判断

则优先走 `answer`。

## Rule 4：批量整理/修补优先走 `organize`

如果用户主要动作是：

- “整理”
- “补齐”
- “更新这一批”
- “把这堆知识合并”

则优先走 `organize`。

## Rule 5：多意图请求内部拆分

例如：

> “先回答这个问题，如果没有就帮我登记缺口，再顺手整理相关知识。”

系统内部应拆成：

1. `answer`
2. gap / write-back
3. `organize`

而不是要求用户改写输入。

---

## 缺口自动触发策略

系统不要求用户自己说“请帮我 write-back”。  
以下情况自动触发缺口动作：

1. 用户提问，但 canon 无覆盖
2. canon 只有部分覆盖，且回答中出现明确缺口
3. 相关页面存在冲突，但没有裁决机制
4. 吸收资料时发现现有 canon 没有对应页面

自动触发的动作默认是：

- 起草 proposal
- 填充 schema
- 写入系统动作摘要

默认**不自动直接写 canon**。

---

## 高风险动作确认门

以下动作必须进入确认门，而不是完全自动执行：

1. 直接影响 canon 现有稳定结论的更新
2. 需要在冲突知识之间做裁决
3. 涉及低置信 source 的重大覆盖
4. 批量影响多个关键页面的合并/替换

确认门只暴露一个最小问题：

- 是否执行
- 为什么要确认
- 如果不执行会怎样

---

## 默认输出与路由的关系

### `answer` 默认输出

- 结论
- 依据
- 缺口 / 争议
- 系统动作摘要

### `absorb` 默认输出

- 已吸收什么
- 生成了哪些 source / proposal
- 是否需要人工确认

### `organize` 默认输出

- 做了哪些整理动作
- 哪些知识被补齐
- 是否有高风险变更待确认

### `audit` 默认输出

- 内部路由链路
- 使用的证据
- proposal 位置
- 风险拦截点

---

## 路由失败时的默认策略

如果系统无法明确判断用户意图：

1. 优先按 **answer-first** 处理
2. 如果 answer 中暴露明显缺口，再自动补 write-back
3. 仅在确实无法安全执行时才追问用户

原因：

- 这最符合“效果优先”
- 也最符合“少打扰用户”

---

## 非目标

本阶段不追求：

1. 训练一个复杂的黑箱分类器
2. 用大量菜单让用户自己选路由
3. 让每条内部路由都单独暴露成命令

---

## 成功标准

如果满足以下条件，则说明路由设计合格：

1. 用户多数请求可以不追问直接路由
2. 路由错误不会直接导致高风险知识写入
3. 用户不需要主动理解系统内部动作边界
4. 自动 write-back 可以稳定接在 `answer` / `absorb` 之后

---

## 当前结论

单入口 Skill 的成败，不取决于“是不是一个入口”，而取决于：

> **入口背后的路由是否足够聪明，而且不会把复杂度重新丢回给用户。**

这份设计的作用，就是把“自动路由”从口号变成真正可实现的内部机制。
