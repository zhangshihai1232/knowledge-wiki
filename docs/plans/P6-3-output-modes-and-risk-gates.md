# P6-3：默认输出模式与风险确认门设计

## 背景

单入口 Skill 是否“低认知负担”，不取决于入口名字，而取决于两件事：

1. **默认输出是不是足够短、足够清楚**
2. **系统什么时候会打断用户确认**

如果输出太技术化，用户还是会被拖进系统内部。  
如果确认过多，用户还是会觉得麻烦。  
如果确认过少，又会带来风险。

所以必须把“默认输出模式”和“风险确认门”一起设计。

---

## 目标

定义一套：

1. **默认极简**
2. **必要时才展开**
3. **高风险时必须拦截**
4. **不牺牲效果**

的前台交互规则。

---

## 设计原则

### 原则 1：默认只给用户最有用的信息

默认输出不是系统日志。  
默认输出必须围绕：

- 结果
- 边界
- 动作摘要
- 必要确认

### 原则 2：解释权按需打开

只有在用户明确要求时，才展开：

- 命中页面
- 路由过程
- 外推断句子
- proposal 位置

### 原则 3：把高风险动作和低风险动作明确分层

低风险动作自动做。  
高风险动作必须进确认门。

---

## 默认输出模式（Default Mode）

## 统一输出框架

默认模式固定为 4 段：

1. **结果**
2. **边界**
3. **系统动作**
4. **需要你决定**

### 模板

```text
结果：{系统给出的最终结果}
边界：{哪些有依据 / 哪些仍有缺口或争议}
系统动作：{是否登记 proposal / 是否触发后台维护}
需要你决定：{无 / 一个明确决策点}
```

### 特点

- 不暴露内部术语
- 不默认展示 proposal 路径
- 不默认展示 schema 字段
- 不默认展示 benchmark / protocol

---

## 审计输出模式（Audit Mode）

只有在用户明确要求时，才进入审计模式。

## 审计模式展开内容

1. 命中的 canon 页面
2. 哪些句子属于外推断
3. proposal 写入位置
4. 内部路由结果
5. 哪个规则拦截了高风险动作

### 审计模式模板

```text
结果：...
依据页面：...
外推断句子：...
内部路由：...
proposal 位置：...
风险拦截：...
```

---

## 三类用户请求的默认输出

### 1. 问答类

默认输出：

1. 直接答案
2. 是否存在缺口 / 争议
3. 是否自动登记了 write-back
4. 是否需要确认

### 2. 摄入类

默认输出：

1. 已吸收什么资料
2. 生成了哪些 source / proposal
3. 是否进入待审
4. 是否需要确认

### 3. 整理类

默认输出：

1. 做了哪些整理动作
2. 哪些知识被补齐 / 对齐
3. 是否触发后台维护
4. 是否存在高风险变更待确认

---

## 风险分层模型

把动作分为 3 层：

| 风险层级 | 系统策略 | 用户负担 |
|---|---|---|
| **R1 低风险** | 自动执行，不打断用户 | 最低 |
| **R2 中风险** | 自动执行，但在结果中提示 | 低 |
| **R3 高风险** | 必须进入确认门 | 中 |

### R1 低风险动作

包括：

- query 回答
- 缺口检测
- write-back proposal 起草
- schema 自动补全
- lint / maintain 后台运行

策略：

- 自动执行
- 只在“系统动作”中简要告知

### R2 中风险动作

包括：

- 大批量整理但不直接写 canon
- 批量 proposal 生成
- 多页健康修复建议

策略：

- 默认执行
- 结果中明确提示已做什么
- 不主动弹确认框

### R3 高风险动作

包括：

1. 直接影响 canon 稳定结论的更新
2. 冲突知识裁决
3. 低置信 source 覆盖高置信内容
4. 大范围合并 / 替换关键页面

策略：

- 不自动执行
- 必须进入确认门

---

## 确认门设计

确认门必须满足两个条件：

1. 问题足够少
2. 风险足够清楚

### 确认门模板

```text
系统发现一个高风险动作：{动作描述}
原因：{为什么这是高风险}
影响：{如果执行，会影响什么}
请选择：执行 / 暂缓
```

### 禁止的确认门坏味道

1. 一次弹出多个技术问题
2. 让用户去理解 schema 字段
3. 让用户决定内部路由
4. 把系统应该自动完成的动作交回给用户

---

## 最小打扰规则

系统只有在以下情况才应打断用户：

1. 进入 R3 高风险动作
2. 用户输入意图确实无法安全判断
3. 现有资料不足以继续执行，且无法内部降级

除此之外，默认不打扰。

---

## 非目标

本设计不追求：

1. 对所有内部动作都弹确认
2. 把所有系统细节默认展示给用户
3. 用“可解释性”换取高认知负担

---

## 成功标准

如果满足以下条件，就说明前台交互设计合格：

1. 默认输出可以一眼看懂
2. 用户大多数时候不需要二次追问“系统刚才做了什么”
3. 高风险动作能被稳定拦住
4. 默认模式不把用户拖进系统术语
5. 审计模式仍然能让高级用户看到完整细节

---

## 当前结论

低认知负担不是“少给信息”，而是：

> **只给当前用户真正需要的信息，把系统复杂性压到后台，只在高风险时把决策点抬出来。**

这份设计定义的，就是这个前台交互边界。
