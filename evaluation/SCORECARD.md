## 各属性详细分析

### 属性1：知识提取保真度 — 6/10

**评分依据**：ingest spec 对声明提取有明确的语义保真要求，原文存档机制完整。但 QG-2 依赖 AI 自我目视检查，缺乏可自动化验证的机制；声明粒度上限（10条）在信息密度高的资料场景下会强迫 AI 合并知识点，与保真目标相悖。

**✅ 优势**
- 声明提取规则明确禁止"合并不同来源内容"、"改变命题方向"，语义保真要求清晰
- source-page.md 要求 `## 原始内容` 保留原始内容，不改写、不摘要，原文存档机制完整
- 每条声明需注明"来源段落"，提供段落级追溯基础
- 升级规则要求矛盾声明保留双方，不静默覆盖，防止提取阶段隐式改写

**❌ 问题**
- **[CRITICAL]** QG-2 是"目视检查"，无可执行的自动化验证机制。AI 无法可靠检测自己是否改写了原文，形成循环依赖
- **[HIGH]** 声明粒度 3-10 条上限在信息密度高的资料场景下会强迫 AI 合并或省略知识点，与"不得合并"原则直接矛盾
- **[HIGH]** "来源段落"定义模糊（段落编号？段落标题？引用原文前N字？），不同执行实例会产生不一致的引用格式
- **[LOW]** 未定义当资料本身存在不确定性表达时（如"可能"、"据悉"）如何在声明中保留原有语气

**→ 改进方向**：QG-2 增加 checksum 验证（`## 原始内容` 节哈希与用户上传原文比对）；声明数量上限改为"软上限+强制理由"；明确"来源段落"引用格式规范

### 属性2：来源可追溯性 — 7/10

**评分依据**：canon→proposal→source 三层追溯链设计完整，compile Gate 1 强制 sources 非空，trigger_source 字段（必填）确保每个 proposal 有明确来源指向。主要风险在于兜底机制会在特定场景破坏追溯链，且 Gate 1 不检查路径有效性。

**✅ 优势**
- compile Step 3 明确要求 sources 字段追加新来源、去重、保留原有来源，累积式来源记录设计合理
- compile Gate 1 断言 `len(canon_page.frontmatter.sources) > 0`，防止空 sources 通过编译
- change-proposal.md 的 `trigger_source` 字段（必填）形成 canon→proposal→source 的中间链
- source-page.md 包含可选的 `checksum`（SHA-256），支持去重和文件完整性验证
- lint L003（ERROR 级别）作为兜底，检测 sources 为空的 canon 页面

**❌ 问题**
- **[HIGH]** 兜底机制（使用 proposal 路径代替 source 路径）会破坏追溯链：若 proposal 被归档或移动，追溯链断裂，且这种断裂不会被 L003 检测到（sources 非空但路径类型错误）
- **[HIGH]** compile Gate 1 只检查 sources 非空，不检查路径有效性。一个失效路径也能通过 Gate 1，需等到 lint 才能发现——但 L004 检查的是 cross_refs 而非 sources 路径，存在检测盲区
- **[MEDIUM]** 从 source 文件到具体声明的追溯粒度不足：canon 页面只能追溯到 source 文件，无法直接追溯到 source 文件中的具体声明
- **[LOW]** sources 字段去重逻辑未定义去重粒度（按文件路径还是按内容哈希）

**→ 改进方向**：限制兜底机制适用范围，用特殊标记区分"待核实来源"；Gate 1 增加路径存在性检查；新增 lint 规则检查 sources 路径有效性

### 属性3：导航可达性 — 6/10

**评分依据**：compile Step 5 覆盖五类 action，首次创建领域有明确的自动初始化规则，lint L001/L009/L004 提供事后检查与自动修复。但 split/merge 操作的源条目清理规则缺失，L001 自动修复存在语义语境盲区，这些漏洞累积会在非正常路径下产生孤立页面。

**✅ 优势**
- compile Step 5 定义了 create/archive/update/merge/split 五类 action 对应的 MOC 更新逻辑，覆盖页面生命周期主要转折点
- 首次创建领域时有完整的自动初始化规则：`_index.md` 不存在时自动创建，并在顶层索引追加领域条目
- lint L001（孤立页面）、L009（缺失索引）均提供自动修复能力
- 分类区块的兜底逻辑清晰：若目标分类区块不存在，在文件末尾新建，不会因缺少区块而失败

**❌ 问题**
- **[CRITICAL]** split 操作后原 slug 的 MOC 清理规则缺失：split 后原 slug 若仍保留在 `_index.md` 中，将导致导航指向已不存在的条目，形成断裂引用（L004 ERROR）
- **[CRITICAL]** L001 自动修复存在语义盲区：孤立状态可能因"应归档未执行"产生，自动修复会将不应出现在导航中的页面强行注入 MOC
- **[HIGH]** merge 操作对被合并方的 MOC 清理未提及，可能留下指向已不存在页面的死链条目
- **[HIGH]** 顶层 `_index.md` 追加领域条目缺乏幂等性保证，异常重跑 compile 可能导致领域条目被重复写入

**→ 改进方向**：在 compile Step 5 中为 split 和 merge 补充"源条目清理"规则；L001 自动修复降级为建议操作，生成候选列表供人工确认；追加条目前增加幂等检查

### 属性4：知识关联准确性 — 5/10

**评分依据**：`[[slug]]` 扫描机制能准确捕捉显式关联，L004 + maintain 修复流程建立了"有则准确"的保证。但系统完全缺乏"应有未有"的发现机制，关联双向性未被系统保证，整体关联质量高度依赖作者意识。

**✅ 优势**
- compile Step 4 通过扫描 `[[slug]]` 自动生成 cross_refs，将显式 wiki link 转为结构化字段
- 对指向不存在 canon 页的引用保留但记录 WARN，保留追溯线索而非简单丢弃
- lint L004（broken-ref ERROR）能发现断裂的 cross_refs
- maintain spec 的 cross-ref 修复提供了按优先级处理的结构化流程（路径更新→归档注释→移除）

**❌ 问题**
- **[CRITICAL]** cross_refs 生成完全依赖显式 wiki link，隐性关联（语义相关但无显式引用）无法被捕获
- **[CRITICAL]** 缺乏"应关联但未关联"的发现机制：无基于领域的同域关联推荐，无基于语义相似度的跨域关联提示
- **[HIGH]** 关联的双向性未被系统保证：A 的 cross_refs 包含 B，但 B 未包含 A，设计中无双向一致性检查规则
- **[HIGH]** 新页面创建时无反向引用初始化：旧页面预先引用了未来 slug 的情况，无法被新页面的 cross_refs 捕获
- **[LOW]** cross_refs 关联权重缺失，无法区分"核心概念引用"与"顺带提及"

**→ 改进方向**：新增 lint 双向一致性检查规则；compile create action 中增加反向引用扫描；为领域内从未被引用的页面产生 INFO 提示

### 属性5：冲突发现与解决 — 5/10

**评分依据**：系统在冲突"发现"层面有基础覆盖（三类矛盾定义 + `<<<CONFLICT>>>` 标记 + lint L006 双路检测），但在冲突"解决"层面存在根本性空白：reconcile 规范缺失、责任人未定义、冲突 proposal 无退出路径，发现→解决未形成闭环。

**✅ 优势**
- ingest 升级规则清晰列举了"事实相反、数值不同、明确否定/修正"三类矛盾定义
- compile 定义了标准 `<<<CONFLICT>>>` 标记格式，冲突可视化
- 冲突检出时自动将 confidence 降为 low，为后续处理留出标记
- lint L006 能覆盖"正文已有冲突标记"和"跨页相同事实不一致断言"两种形式
- proposal 处理时要求同时引用现有内容和新声明，双方证据均保留

**❌ 问题**
- **[CRITICAL]** 冲突解决责任人完全模糊：ingest 只说"需人工判断"，compile 更是"不自动裁决"后无后续，无人认领导致冲突堆积
- **[CRITICAL]** 缺少"resolve conflict"规范：系统有冲突识别机制，但没有明确的 reconcile spec，冲突 proposal 停留在 inbox 无出口
- **[HIGH]** L006 跨页矛盾检测可操作性差：要求"同一实体的同一属性的不一致表述"触发告警，但未定义"实体-属性"的标准化表达形式，难以自动化
- **[HIGH]** 冲突升级规则仅在 ingest 阶段触发：canon 中两个已有页面后来产生矛盾，系统无法主动发现，只能依赖定期 lint
- **[MEDIUM]** 冲突来源权重未定义：新资料与 canon 冲突时，系统保留双方但不指示哪一方更可信

**→ 改进方向**：新增 reconcile.md 定义冲突解决完整流程；冲突 proposal 自动路由到专属 review 队列；在 canon-page.md schema 中增加 `facts` 字段支持 L006 自动化检测

### 属性6：置信度准确性 — 4/10

**评分依据**：confidence 字段的机制设计过于单薄：唯一有明文规范的更新是"冲突时自动降为 low"，初始值设置规则和升级路径均未定义。低置信页面被标记后无法恢复，confidence 字段在实际使用中只能单向恶化。

**✅ 优势**
- compile 检出冲突时自动将 confidence 降为 low，体现了对矛盾内容可信度的惩罚机制
- lint L005 能识别哪些页面长期处于低置信状态
- proposal 中 confidence 字段（必填）提供了初始置信度的记录入口

**❌ 问题**
- **[CRITICAL]** confidence 唯一的更新触发是冲突降级，升级路径完全缺失：一旦降为 low，无标准化的升级条件
- **[CRITICAL]** 初始置信度设置无规则：由谁决定？基于什么标准？LLM 判断？用户指定？来源权重？完全空白
- **[HIGH]** confidence 与 staleness_days 的关系不清：两者的相互影响规则未定义，一个页面可能 staleness_days=0 但 confidence=low
- **[HIGH]** 缺少来源质量与 confidence 的映射：多个 sources 的可信度差异是否应影响 canon 页的 confidence？规范中完全未提及
- **[HIGH]** L005 驱动力过弱：触发条件是 INFO 级（最低级别），建议是"人工审核"，结果往往是告警被忽视

**→ 改进方向**：在 source frontmatter 中增加 `authority` 字段，proposal 继承最高来源权威性作为初始 confidence；定义 low→medium→high 的显式升级条件；L005 改为 WARNING 级别

### 属性7：陈旧知识处理机制 — 4/10

**评分依据**：系统设计了合理的多级阈值（30/90/180天）和对应 lint 规则，但存在根本性缺陷：`staleness_days` 的自动增长机制完全未定义，导致整套阈值检测在实践中无法自动触发。此外 lint L002 发现陈旧页面到 maintain 执行归档之间存在路径断层，refresh spec 完全缺失。

**✅ 优势**
- 三级阈值分层清晰：30天（L005低置信预警）→ 90天（L002陈旧警告）→ 180天（归档候选），梯度合理
- L005 对低置信页面额外保护：confidence=low 的页面在 30 天时就预警，比普通页面早60天
- 归档触发条件双重门禁：staleness>180天 且 无近期引用，避免误归档高价值历史页面
- 归档不等于删除：status:archived 保留文件，仅从 `_index.md` 移除，满足审计追溯需求
- 禁止归档 status:pinned 页面，对不同类型页面有差异化保护

**❌ 问题**
- **[CRITICAL]** `staleness_days` 计时机制根本缺失：compile 负责重置为0，但整个设计文件中没有任何地方定义它如何随日历时间自动增长，L002/L005 的阈值检测在实践中永远无法自触发
- **[CRITICAL]** refresh spec 完全缺失：单条 L002 的处理路径（"建议触发 compile 或维护者确认"）没有对应规范，形成路径断点
- **[HIGH]** "无近期引用"条件可操作性不足：未定义"近期"指多少天，系统如何追踪页面被引用的事件，该条件在现有 schema 中无法被客观评估
- **[HIGH]** 单条 L002 与 maintain 的触发缝隙：≤2条 L002 不满足 maintain 触发条件，refresh spec 又不存在，这些页面处于"被发现但无人处理"的灰色地带

**→ 改进方向**：在 lint spec 中明确 `staleness_days = (lint_run_date - last_compiled).days`，使其成为动态计算值；定义 refresh spec 处理单条 L002；在 canon-page.md 增加 `last_referenced_date` 字段量化"近期引用"

### 属性8：知识库活跃度 — 3/10

**评分依据**：系统整体是被动响应式设计，知识库的改善完全依赖外部事件触发（新资料摄入、人工维护指令）。STATE.md 记录的指标是静态计数，无法反映知识质量趋势；confidence 分布和归档页面数均未被追踪，系统无法自知是否在变好或变坏。

**✅ 优势**
- L007 领域溢出 → 领域分裂的结构自愈机制：单一领域超过50页时自动预警，是少数几个主动防止退化的设计之一
- confidence 字段的质量标注提供了页面级质量管理的基础
- compile 闭环带来的局部质量提升：每次 compile 将多个 sources 整合进 canon，理论上覆盖来源越广质量越高
- archive 机制防止知识库被低质量旧内容拖累

**❌ 问题**
- **[CRITICAL]** STATE.md 指标不足以监控健康趋势：缺失 confidence 分布（high/medium/low 各占比）、平均 staleness_days、归档速率 vs 新建速率、open_conflicts 趋势等关键指标，系统无法判断自己是在变好还是退化
- **[CRITICAL]** 系统缺乏"主动知识升级"驱动力：没有任何流程会主动发起"这个页面应该被重写或升级"，低置信页面一旦产生可能永久停留在 low 状态
- **[CRITICAL]** confidence 字段缺乏生命周期管理：初始值如何确定、低置信如何升级、高置信在什么条件下降级，三个问题均无答案，字段只能单向恶化
- **[HIGH]** 没有"用户使用→知识改善"的正向飞轮：用户发现错误信息、过时内容、缺失知识点，都没有标准路径将反馈转化为知识更新动作
- **[HIGH]** 领域分裂后缺乏效果追踪：分裂后没有机制验证"分裂是否解决了原有问题"，分裂可能成为形式操作

**→ 改进方向**：将 STATE.md 升级为健康仪表板，增加 confidence_distribution、avg_staleness_days、archive_rate 等趋势指标；定义 confidence 升级路径；引入轻量级 feedback spec 建立使用反馈通道

### 属性9：人工审查有效性 — 6/10

**评分依据**：promote spec 设计了多层次的审查信息展示（diff、canon 现状、原始证据、AI 建议），信息结构完整，有助于审查者做出有质量的判断。但缺乏强制机制防止"一键批准"，rejection_reason 的质量保证仅停在"非占位符"层面，reopen 机制存在但缺少改进引导。

**✅ 优势**
- Step 2 的四类审查信息层次清晰：diff 看变化量，canon 现状看影响面，原始证据验证来源真实性，AI 建议提供结构化风险评估
- Gate 1 和 Gate 2 作为硬性门禁，确保 reviewed_by/reviewed_at 字段不可缺失，防止无主提案流入 canon
- reopen 机制使被拒提案可以改进后重新提交，保留知识演化的可能性
- 升级规则（Lock 级别）对高风险操作（删除 canon 页、领域合并、>50% 改写）有专门保护
- approve/reject/modify/reopen 四种决策路径支持迭代改进而非简单二元决策

**❌ 问题**
- **[CRITICAL]** 无防止"一键全 approve"的机制：没有强制阅读时间、批量 approve 限制或抽样复核要求，审查者可以在不阅读任何内容的情况下连续 approve 所有待审提案
- **[HIGH]** rejection_reason 质量无法保证：Gate 2 仅排除"占位符"，"内容有误"这类一行模板化理由同样满足 Gate 2，却完全没有指导提案作者如何改进
- **[HIGH]** AI 建议（Step 2d）的质量缺乏校验：AI 建议本身可能误判，但设计中没有要求审查者在不同意 AI 建议时做出说明，AI 建议可能反向引导审查者批量跟随
- **[MEDIUM]** reopen 机制缺少改进说明要求：未修改的提案可以反复 reopen-审查-reject 循环
- **[LOW]** modify 决策路径设计粗糙：没有规定谁来修改、修改后是否需要重新触发 Step 2 流程

**→ 改进方向**：approve 操作增加必填字段 `approve_note`（制造最低限度认知摩擦）；rejection_reason 升级为"必须包含具体问题描述和改进方向"；reopen 操作要求提案内容与上次被拒版本存在实质差异

---

### 属性10：审查负担可控性 — 4/10

**评分依据**：设计中存在积压预警机制（lint L008），但这是被动预警而非主动控制。根本问题在于 ingest 的 autonomy=auto 会无限制地将每次摄入产生的提案推入 inbox，而 promote 的 autonomy=propose 意味着审查完全依赖人工触发，系统在高摄入场景下存在明显的审查瓶颈风险。

**✅ 优势**
- lint L008（stale-proposal WARNING）提供了积压可见性，超过7天未处理时会报警
- promote Step 1 按 proposed_at 升序排列，确保先进先出，防止老提案被淹没
- 升级规则将高风险提案单独标识，审查者可识别需要更多精力的提案

**❌ 问题**
- **[CRITICAL]** 无提案生成速率控制：ingest autonomy=auto 意味着每个摄入文件都可能生成提案，设计中没有提案去重、批量合并或生成频率限制机制
- **[CRITICAL]** 缺乏 AI 预过滤机制：所有 ingest 生成的提案直接进入审查队列，没有对明显低质量提案（信息量极低、与现有 canon 完全重复）进行预筛选
- **[HIGH]** 无批量审查设计：promote 流程是逐个处理，积压达到20+提案时，缺乏"相似提案批量处理"、"低风险提案快速通道"等效率机制
- **[HIGH]** lint L008 是7天单一 WARNING，对严重积压的响应力不足，无更高级别的积压告警
- **[MEDIUM]** query write-back 也会生成 inbox 提案，与 ingest 提案叠加后审查压力更大，但设计中未区分两类提案的优先级

**→ 改进方向**：ingest 阶段增加提案质量预评分，低于阈值进入暂存区；增加提案去重机制；为低风险提案设计"快速通道"；升级 L008 为三级积压告警（WARNING/ERROR/CRITICAL）

### 属性11：知识缺口主动发现 — 5/10

**评分依据**：query write-back 机制提供了一个被动的缺口发现路径（由查询触发），lint 的10条规则覆盖了结构性问题。但系统完全缺乏主动扫描机制，依赖用户查询触发，无法发现"没人问过但很关键"的知识盲区。

**✅ 优势**
- query write-back 的三个触发条件设计合理：完全无覆盖、缺少具体概念、大量 canon 外推断，覆盖从"领域缺失"到"局部概念缺失"的不同粒度
- write-back 生成的 proposal 文件名包含日期和 slug 建议，具有良好的可追溯性
- trigger_source 字段在 write-back 场景中保留了知识缺口的来源上下文，有助于后续审查
- lint L003、L004 等规则捕捉了来源和引用的结构性缺陷

**❌ 问题**
- **[CRITICAL]** 缺口发现完全依赖用户查询触发：从未被查询的领域（新引入的重要领域、冷门专题）的缺口永远不会被发现，系统对"静默盲区"没有任何探测能力
- **[HIGH]** write-back 生成的 proposal 质量缺乏保证：proposal 内容是 AI 推断应补充什么，没有要求包含候选内容草稿或参考来源，审查者收到的可能只是"这个领域需要补充"，无法直接推动知识生产
- **[HIGH]** lint 10条规则仅覆盖结构性缺陷，不包含任何内容层面的缺口检测：canon 页存在但内容陈旧、覆盖某领域但深度严重不足等问题均无法通过 lint 发现
- **[MEDIUM]** write-back 触发条件"大量 canon 外推断"的"大量"未量化，触发实际上依赖 AI 自我判断，可能存在漏报
- **[LOW]** 没有缺口提案的聚合分析机制：多次查询触发的 write-back 可能指向同一个知识盲区，但每次生成独立提案，无法识别高频缺口

**→ 改进方向**：在 maintain 中增加主动扫描任务（定期比对 source 与 canon 的覆盖关系）；要求 write-back 提案包含 `candidate_content` 字段；明确 write-back 触发的量化阈值（如推断比例>25%）

### 属性12：查询回答可信度标注 — 7/10

**评分依据**：query.md 定义了完整的三级可信度标注体系，Gate 2 要求无相关知识时明确告知用户，标注机制的设计在原则层面是扎实的。主要风险在于"合理推断"界定模糊、Gate 的可执行性依赖 AI 自律，以及 slug 格式对用户的可访问性未作设计。

**✅ 优势**
- 三级标注体系覆盖完整的可信度谱系：直接引用（`[来源: slug]`）→ 有据推断（`[⚠️ canon外推断]`）→ 无依据（移除或转入知识缺口说明）
- Gate 2 明确规定 canon 完全无相关内容时必须告知用户，禁止伪装成 canon 内容输出，是防止系统"说谎"的最重要保障
- Step 4 的来源完整性检查要求每个事实性声明必须映射到至少一个具体 canon 页 slug，标注粒度是声明级而非段落级
- 对"直接内容"与"合理推断"的区分处理体现了对知识来源不同可信度的尊重

**❌ 问题**
- **[HIGH]** "合理推断（非直接引用）"的界定标准缺失：未说明什么情况下算"合理推断"而非"训练知识补充"，这两者在 AI 执行层面极难区分，可能导致大量训练知识以"合理推断"名义混入回答
- **[HIGH]** Gate 1（事实性声明100%带来源标注）的可执行性依赖 AI 自我遵守：没有外部校验机制，违反 Gate 1 的回答无法被自动发现
- **[MEDIUM]** "事实性声明"的范围界定不明确：比较性陈述、评价性陈述、条件性陈述是否都需要标注，设计中没有给出判断标准
- **[MEDIUM]** 标注格式 `[来源: slug]` 对用户可用性存疑：slug 是系统内部标识符，普通用户看到 `[来源: concepts/vector-index]` 不知道如何访问验证
- **[LOW]** Step 5 write-back 与 Step 4 标注动作之间关系不清晰：触发 write-back 时，回答本身是否仍需包含 canon 外推断标注，两个步骤的边界未明确

**→ 改进方向**：增加"合理推断"的操作性定义（推断必须有 canon 中的间接支撑）；在 query 流程结束时增加自检步骤（AI 扫描回答统计未带标注的陈述句）；将 slug 转换为可访问路径格式
