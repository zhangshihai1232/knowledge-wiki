# 知识治理质量评估框架

## 设计原则

本框架从知识治理的本质出发，评估一套知识管理系统是否能：
1. 可靠地从原始资料中提取知识
2. 让知识质量随时间积累而提升
3. 在查询时提供有来源、可信的回答
4. 保持人工审查负担可控
5. 支持知识库的持续演化

## 16 个质量属性

### 第一层：知识进入

#### 属性 1：知识提取保真度（Knowledge Extraction Fidelity）

**核心问题**：从原始资料到声明，有没有失真、改写、遗漏？

**为什么重要**：Ingest 是知识进入系统的唯一入口。提取阶段的失真会污染整个知识库。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 声明严格保持原文语义，有明确的段落引用，粒度合理（3-10条/资料），无改写 |
| 7-8 | 绝大多数声明保真，极少量有轻微概括但不改变语义 |
| 5-6 | 部分声明存在语义概括或遗漏重要细节 |
| 3-4 | 多处声明存在改写或观点偏移 |
| 0-2 | 声明与原文严重不符，或系统允许改写原始内容 |

**评估方法**：抽样 10-20 个 source 文件，对比 `## 提取声明` 与 `## 原始内容` 的语义一致性

**对应设计**：ingest Step 3、QG-2（正文未改写）

---

#### 属性 2：来源可追溯性（Source Traceability Completeness）

**核心问题**：每条 canon 知识都能追溯到原始 source 吗？

**为什么重要**：Canon 知识的可信度完全依赖来源追溯。无来源 = 无法验证 = 不可信。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 所有 canon 页面 sources 非空，每条 source 路径都指向存在的文件，追溯链完整 |
| 7-8 | sources 均非空，极少量路径失效（< 2%） |
| 5-6 | 少量页面 sources 为空或路径失效（2-10%） |
| 3-4 | 较多页面 sources 缺失（10-30%） |
| 0-2 | sources 机制形同虚设，大量页面无法追溯来源 |

**评估方法**：lint L003 零容忍；抽样验证 source 路径有效性；从 canon 页追溯到 source 的提取声明节

**对应设计**：compile Step 3、lint L003

---

### 第二层：知识结构

#### 属性 3：导航可达性（Navigation Reachability）

**核心问题**：知识能被找到吗？孤立页面和断裂引用是否为零？

**为什么重要**：存在但无法被导航到的知识等于不存在。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 零孤立页面，零断裂引用，所有领域有完整 _index.md |
| 7-8 | 孤立页面 < 2%，断裂引用 < 1% |
| 5-6 | 孤立页面 2-10%，或有领域缺少 _index.md |
| 3-4 | 孤立页面 > 10%，导航结构残缺 |
| 0-2 | 大量页面无法通过导航到达 |

**评估方法**：lint L001（orphan-page）、L004（broken-ref）、L009（missing-index）计数

**对应设计**：compile Step 5、lint L001/L004/L009

---

#### 属性 4：知识关联准确性（Cross-reference Accuracy）

**核心问题**：cross_refs 是否真实反映了知识间的语义关系？

**为什么重要**：不准确的关联会误导查询，让用户在无关页面间跳转。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 关联均有语义依据，无冗余关联，重要关联未缺失 |
| 7-8 | 关联基本准确，少量冗余或遗漏 |
| 5-6 | 存在明显的无关联或重要关联缺失 |
| 3-4 | 关联混乱，大量无意义引用 |
| 0-2 | cross_refs 机制未被有效使用 |

**评估方法**：抽样审核 cross_refs 的语义相关性；检查是否存在"应关联但未关联"的页面对

**对应设计**：compile Step 4、maintain cross-ref 修复

---

### 第三层：知识一致性

#### 属性 5：冲突发现与解决（Conflict Detection & Resolution）

**核心问题**：矛盾知识是否被标记？是否在合理时间内被解决？

**为什么重要**：未解决的冲突意味着知识库对同一事实有两种不同答案，严重损害可信度。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 有冲突检测机制，冲突 30 天内解决率 > 80%，解决后保留双方证据 |
| 7-8 | 冲突能被发现并标记，解决周期 30-60 天 |
| 5-6 | 冲突能被标记但长期积压，解决率 < 50% |
| 3-4 | 冲突检测机制存在但实际未被执行 |
| 0-2 | 无冲突检测，矛盾知识静默共存 |

**评估方法**：lint L006 计数；STATE.md open_conflicts 趋势；冲突标记平均存续天数

**对应设计**：ingest 升级规则、compile 冲突检测、lint L006

---

#### 属性 6：置信度准确性（Confidence Level Accuracy）

**核心问题**：confidence 值是否真实反映了知识的可信程度？

**为什么重要**：错误的 confidence 会让用户对低质量知识过度信任，或对高质量知识不必要地怀疑。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | confidence 值与 sources 数量、冲突状态、staleness 高度相关；低置信页面被及时处理 |
| 7-8 | confidence 基本准确，少量标注偏差 |
| 5-6 | confidence 值随意设置，与实际质量相关性弱 |
| 3-4 | 大量页面 confidence 与实际质量明显不符 |
| 0-2 | confidence 机制形同虚设 |

**评估方法**：抽样验证 high-confidence 页面是否有充足来源；检查 low-confidence 页面的处理率；lint L005 超期数量

**对应设计**：compile confidence 更新规则、lint L005

---

### 第四层：知识时效性

#### 属性 7：陈旧知识处理机制（Knowledge Freshness Maintenance）

**核心问题**：过时知识是否被发现并归档或更新？

**为什么重要**：知识会过时。不处理陈旧知识会让用户依赖错误信息。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 陈旧页面（staleness > 90天）< 5%，发现后 30 天内处理率 > 80% |
| 7-8 | 陈旧页面 5-15%，有处理机制但执行不够及时 |
| 5-6 | 陈旧页面 15-30%，处理机制存在但执行率低 |
| 3-4 | 陈旧页面 > 30%，知识库逐步退化 |
| 0-2 | 无时效性管理机制，知识库持续老化 |

**评估方法**：lint L002 计数；staleness_days 分布统计；归档操作执行频率

**对应设计**：lint L002、maintain 内容归档

---

#### 属性 8：知识库活跃度（Repository Vitality）

**核心问题**：知识质量是在随时间提升还是退化？

**为什么重要**：这是对系统整体承诺的验证——知识编译器应该让知识越来越好，而不是越来越乱。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | high-confidence 页面比例持续上升，陈旧页面比例稳定或下降，新增页面质量高 |
| 7-8 | 整体趋势向好，偶有波动 |
| 5-6 | 质量指标基本持平，无明显改善 |
| 3-4 | 质量指标缓慢下滑 |
| 0-2 | 知识库明显退化，陈旧和冲突积累速度超过处理速度 |

**评估方法**：月度统计 high/medium/low confidence 比例变化趋势；新增 vs 归档页面比；lint score 历史趋势

**对应设计**：系统整体长期运行效果

---

### 第五层：治理过程

#### 属性 9：人工审查有效性（Promote Review Effectiveness）

**核心问题**：审查是否真正筛选了质量？拒绝率和理由是否有实质内容？

**为什么重要**：promote 是防止错误知识进入 canon 的唯一关卡。无效审查 = 关卡形同虚设。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 审查覆盖率 100%，拒绝率 5-20%，rejection_reason 具体有实质内容，reopen 成功率 > 60% |
| 7-8 | 审查覆盖率 100%，拒绝率合理，但理由质量参差不齐 |
| 5-6 | 审查覆盖率 100% 但拒绝率 < 3%（形式审查），或理由模板化 |
| 3-4 | 审查覆盖率 < 100%，或存在无 reviewed_by 的 approved 提案 |
| 0-2 | 审查机制被绕过，或大量提案未经审查直接进入 canon |

**评估方法**：统计 approved 提案中 reviewed_by 覆盖率；计算拒绝率；抽样审核 rejection_reason 质量

**对应设计**：promote Step 2-3、QG-1/QG-2

---

#### 属性 10：审查负担可控性（Review Bottleneck Prevention）

**核心问题**：promote 是否成为瓶颈？积压是否超过健康阈值？

**为什么重要**：审查瓶颈会堵塞整个知识管道，让系统在实际使用中失效。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 积压提案 < 5 个，无超期（> 7天）提案，平均处理周期 < 2 天 |
| 7-8 | 积压 5-10 个，超期提案 < 2 个，处理周期 2-5 天 |
| 5-6 | 积压 10-20 个，或有 3-5 个超期提案 |
| 3-4 | 积压 > 20 个，或超期提案 > 5 个，审查明显跟不上摄入速度 |
| 0-2 | 审查严重积压，系统实际上已停止运转 |

**评估方法**：STATE.md pending_proposals；lint L008 超期提案数；ingest 速度 vs promote 速度比

**对应设计**：promote、lint L008、STATE.md pending_proposals

---

### 第六层：知识可用性

#### 属性 11：知识缺口主动发现（Proactive Gap Discovery）

**核心问题**：系统能主动发现自己不知道什么吗？write-back 是否有效闭环？

**为什么重要**：被动等待用户发现缺口会让知识库长期存在盲区。主动发现是知识库自我完善的核心机制。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | query write-back 机制运转良好，缺口 proposal 转化率 > 70%，lint 问题发现率高 |
| 7-8 | write-back 有使用，转化率 50-70% |
| 5-6 | write-back 机制存在但使用率低，缺口大量积压在 inbox |
| 3-4 | write-back 形同虚设，缺口只能靠用户反馈发现 |
| 0-2 | 无主动发现机制 |

**评估方法**：统计 write-back 生成的 proposal 数量及最终 compiled 比例；lint 规则覆盖率

**对应设计**：query Step 5、lint 全部规则

---

#### 属性 12：查询回答可信度标注（Query Answer Trustworthiness）

**核心问题**：每条回答的来源都标注了吗？canon 外推断是否诚实标记？

**为什么重要**：用户无法判断没有标注的信息是否可信。标注是知识治理系统与普通 LLM 的核心区别。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 所有事实声明有 [来源: slug] 标注，所有推断有 [⚠️ canon外推断] 标注，无来源的信息不输出 |
| 7-8 | 来源标注率 > 95%，推断标注率 > 90% |
| 5-6 | 来源标注率 70-95%，部分推断未标注 |
| 3-4 | 来源标注率 < 70%，标注形同虚设 |
| 0-2 | 无来源标注，回答与普通 LLM 无区别 |

**评估方法**：抽样 20-30 个 query 回答，统计标注覆盖率；检查推断标注的诚实性

**对应设计**：query Step 4、QG-1/QG-2

---

### 第七层：系统效能

#### 属性 13：知识利用效率（Knowledge Utilization Efficiency）

**核心问题**：被编译入 canon 的知识是否真正被查询使用？是否存在大量"沉睡知识"？

**为什么重要**：知识编译的最终价值在于被利用。大量未被查询的 canon 页面意味着编译资源浪费，也暗示知识结构可能偏离用户实际需求。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 80%+ 的 canon 页面在 90 天内被查询引用过，query_count 分布均匀，无长期零引用页面 |
| 7-8 | 60-80% 页面有近期引用，少量沉睡页面（< 20%） |
| 5-6 | 40-60% 页面有引用，沉睡页面 20-40% |
| 3-4 | 引用率 < 40%，大量页面从未被查询 |
| 0-2 | 无利用追踪机制，或 canon 页面几乎不被使用 |

**评估方法**：统计 canon 页面的 `last_queried_at` 和 `query_count` 分布；计算 90 天内零引用页面比例；分析高频/低频引用页面的特征差异

**对应设计**：canon-page schema `last_queried_at` / `query_count`、query Step 3

---

#### 属性 14：蒸馏效率（Distillation Efficiency）

**核心问题**：从原始资料到 canon 知识的转化链路是否高效？中间环节是否存在不合理的损耗？

**为什么重要**：蒸馏效率直接决定系统的投入产出比。低转化率意味着大量人工审查和 AI 计算资源被浪费在最终未进入 canon 的提案上。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | source → proposal 转化率 > 80%，proposal → compiled 转化率 > 70%，端到端周期 < 7 天 |
| 7-8 | 端到端转化率合理（50-70%），周期 7-14 天 |
| 5-6 | 转化率 30-50%，或周期 14-30 天，存在明显瓶颈环节 |
| 3-4 | 转化率 < 30%，大量提案积压或被拒，蒸馏链路低效 |
| 0-2 | 蒸馏链路断裂，知识无法有效从 source 流转到 canon |

**评估方法**：统计各环节转化率（ingest 生成 proposal 数 / source 数、compiled proposal 数 / approved 数、approved 数 / inbox 总数）；计算端到端平均周期（proposed_at → compiled_at）

**对应设计**：ingest、promote、compile 全链路、STATE.md 统计字段

---

#### 属性 15：跨领域一致性（Cross-domain Coherence）

**核心问题**：不同领域的 canon 页面在质量标准、结构规范、引用方式上是否保持一致？

**为什么重要**：知识库若各领域质量参差不齐，会导致用户对整体可信度产生怀疑。跨领域引用时的不一致更会直接误导决策。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 各领域 confidence 分布相近（标准差 < 15%），模板遵循率 > 90%，跨域引用准确率 > 95% |
| 7-8 | 领域间质量差异可控（标准差 15-25%），偶有模板偏离 |
| 5-6 | 领域间差异明显（标准差 25-40%），部分领域质量显著低于均值 |
| 3-4 | 领域质量严重分化，部分领域形同虚设 |
| 0-2 | 无跨领域一致性管理，各领域独立演化 |

**评估方法**：按领域统计 confidence 分布、staleness 均值、页面模板遵循率、cross_refs 准确率；计算领域间质量指标标准差

**对应设计**：lint L010（type-path-mismatch）、compile 模板遵循检查、maintain 领域分裂

---

#### 属性 16：系统可演化性（System Evolvability）

**核心问题**：当知识领域扩展、规则需要调整、新场景出现时，系统是否能低成本适应？

**为什么重要**：知识管理是长期工程。系统若缺乏扩展能力，会在知识库规模增长后变得脆弱，最终被废弃。

**评分标准**：

| 分数 | 标准 |
|------|------|
| 9-10 | 新增领域 < 30分钟完成配置，新增 lint 规则不影响现有规则，spec 间耦合度低（修改一个 spec 不需连锁修改 > 2 个其他 spec） |
| 7-8 | 扩展成本可控，偶有跨 spec 连锁修改但可预期 |
| 5-6 | 扩展需要中等成本，部分 spec 耦合度高 |
| 3-4 | 扩展成本高，修改一处经常引发多处不一致 |
| 0-2 | 系统僵化，无法适应新领域或新需求 |

**评估方法**：模拟新增一个领域的完整流程（从 source 到 canon）计时；评估 spec 间依赖图的耦合度；检查 schema 版本机制是否支持向后兼容

**对应设计**：schema version 字段、maintain 领域拆分/合并、lint 规则可扩展架构

## 评估使用说明

### 何时使用本框架

- **设计审查**：在系统落地前，评估设计方案的完整性和可行性
- **版本评审**：系统重大更新后，评估变更是否引入了质量退化
- **定期健康检查**：系统运行 3-6 个月后，评估实际运行质量

### 如何使用 Agent Teams

启动 6+1 个 Agent 并行评估：

```
A1（层1专家）→ 属性1+2 → findings/01-knowledge-ingestion.md
A2（层2专家）→ 属性3+4 → findings/02-knowledge-structure.md
A3（层3专家）→ 属性5+6 → findings/03-knowledge-consistency.md
A4（层4专家）→ 属性7+8 → findings/04-knowledge-freshness.md
A5（层5+6专家）→ 属性9-12 → findings/05-governance-usability.md
A6（层7专家）→ 属性13-16 → findings/06-system-efficiency.md
A0（汇总）→ 读取全部 findings → SCORECARD.md
```

每个 Agent 的 prompt 模板见 `framework/agent-prompts.md`

### 评分说明

- 每个属性独立评分（0-10），**不合并总分**
- 分数反映当前设计/实现的质量，不是对系统的综合判断
- 低分属性是优先改进的方向
- 建议每次评估后对比上次分数，观察趋势
